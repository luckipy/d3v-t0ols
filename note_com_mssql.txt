xp_cmdshell là một extended stored procedure trong Microsoft SQL Server cho phép bạn chạy lệnh hệ điều hành (cmd) trực tiếp từ bên trong SQL Server.

Cách xp_cmdshell hoạt động:
- SQL Server nhận lệnh EXEC xp_cmdshell
- Nó gọi xuống Windows API
- Tạo một process giống như mở cmd.exe
- Thực thi command bạn truyền vào
- Thu output (stdout) và trả về dưới dạng result set


Nó chạy dưới quyền của SQL Server service account.


Stored Procedure là một tập hợp các câu lệnh SQL được lưu sẵn trong database và có thể gọi lại nhiều lần. Nó được viết bằng SQL.
Ví dụ:

    SELECT * FROM Users;

* Login và User trong SQL Server 
** Login (Server login) - (Thẻ căn cước để vào cổng chính)
*** Login tồn tại ở mức Server, tức toàn bộ instance SQL Server. Nó có nhiệm vụ xác thực rằng có được phép kết nối vào SQL Server hay không.
*** Khi thực hiện login vào SQL Server (qua GUI SSMS hoặc command line), nó sẽ kiểm tra Login trước tiên.
**** Nếu sai username/password thì không vào được server
**** Nếu đúng thì đã qua được "cổng chính", nhưng mà chưa chắc vào được các "phòng" (là database)
*** Có 2 loại login phổ biến
**** SQL Login: Dùng username và password dược lưu trong SQL
**** Windows Login: Dùng tài khoản hệ thống Windows (trường hợp này sẽ không cần nhập password và password của nó cũng không được lưu trong SQL)
*** Ví dụ: Khi tạo một tài khoản TestLogin với password là P@ssw0rd123 thì lúc này TestLogin có thể kết nối vào SQL Server nhưng nó chưa có quyền làm gì bên trong bất kỳ database nào cả.

* User (Database User) - (Thẻ ra vào từng phòng  - database)
** User tồn tại ở mức Database (bên trong từng database riêng lẻ)
** Nó có nhiệm vụ ủy quyền rằng bên trong database này thì user được phép làm những gì
** User nó sẽ quyết định thuộc role nào (db_datareader, db_owner...), được select/insert bảng nào, chạy procedure nào...
** Mỗi database có danh sách User riêng, một database có thể có user tên là TestUser, database khác có user tên khác.


* Mapping (Ánh xạ) giữa Login và User – “Kết nối thẻ căn cước với thẻ phòng”
** Một Login (thẻ căn cước) muốn vào một database cụ thể, thì phải có một User trong database đó ánh xạ (map) về Login đó.
** Nếu không có mapping thì dù connect thành công vào server, khi cố gắng USE TenDatabase hoặc chọn database trong SSMS thì sẽ bị lỗi.
** Cách mapping hoạt động:
*** Dựa trên SID (Security Identifier) – một mã số duy nhất.
*** Khi tạo User từ Login: CREATE USER TestUser FOR LOGIN TestLogin;
→ SQL Server tự động lấy SID của Login và gán cho User → chúng được ánh xạ với nhau.
** Một Login có thể map với nhiều User (ở nhiều database khác nhau).
** Một User chỉ map với đúng một Login.
** Ví dụ minh họa:
*** Giả sử có một tài khoản Login: PhucDev (tạo ở mức server)
*** Muốn PhucDev truy cập 2 database:
**** Database HR → tạo User tên PhucHR và map nó với Login PhucDev
**** Database Sales → tạo User tên PhucSales và map nó với Login PhucDev
*** Kết quả:
**** Khi PhucDev connect vào SQL Server → thành công.
**** Khi chọn database HR → nó dùng User PhucHR → được quyền theo role của PhucHR trong HR.
**** Khi chọn database Sales → nó dùng User PhucSales → quyền khác (có thể db_datareader ở Sales, nhưng chỉ public ở HR).
*** Nếu bạn không tạo User trong HR cho login PhucDev → dù login tồn tại, anh ta vẫn không vào được database HR.



* Cần hiểu về Procedure (stored procedure)
** Nó là một đoạn code SQL được lưu sẵn trong database để thực thi lại nhiều lần.
** NÓ là một object trong database chứa: câu lệnh SQL, logic (IF, WHILE, ...) và có thể nhận vào tham số.
** Nếu database là một hệ thống thì:
*** Table là dữ liệu
*** User là người dùng
*** Role là quyền
*** Stored Procedure là “chức năng / hành động được đóng gói sẵn”

** Procedure này nó có tác dụng bảo mật, ngăn user có thể truy cập trực tiếp vào table (vì có thể nhìn thấy toàn bộ dữ liệu của table), nó bắt user phải sử dụng các Procedure được định nghĩa sẵn -> Giới hạn của người dùng nằm trong cách mà Procedure đó xử lý với table.
** Ví dụ: Trong một công ty nó có một bảng dữ liệu gồm: id, username, balance, password.
*** Nếu cấp sai "GRANT SELECT ON Accounts TO app_user;" cho phép người dùng truy cập trực tiếp thì lúc này người dùng chạy "SELECT * FROM Accounts;" thì có thể thấy toàn bộ balance và password của mọi người được lưu trong table đó.
*** Nếu sử dụng Procedure, nó sẽ chặn người dùng đó truy cập trực tiếp vào database trước "DENY SELECT ON Accounts TO app_user;", rồi sau đó định nghĩa một Procedure và cho phép người dùng chạy qua Procedure đó để tương tác với table. "GRANT EXECUTE ON GetMyBalance TO app_user;"
CREATE PROCEDURE GetMyBalance
    @username NVARCHAR(50)
AS
BEGIN
    SELECT balance 
    FROM Accounts 
    WHERE username = @username;
END

- orphan users: các user không có login map (MappedLogin = NULL), thường xảy ra sau khi restore database từ server khác.

impacket-mssqlclient testcontained:testcontained@192.168.17.36 -db TestDB

* Điều kiện bắt buộc để có một Contained Database User
1. Bật Contained Database Authentication ở mức instance (cần quyền sysadmin)
EXEC sp_configure 'contained database authentication', 1; RECONFIGURE;
2. Database phải là Partial Containment (Chỉ Partial mới hỗ trợ contained user.)
ALTER DATABASE YourDB SET CONTAINMENT = PARTIAL; 
hoặc khi tạo DB: CREATE DATABASE YourDB CONTAINMENT = PARTIAL;
3. Tạo user trực tiếp với password trong database (User không được dùng cú pháp FOR LOGIN. Phải là SQL User (không phải Windows).)
CREATE USER ContainedUser WITH PASSWORD = 'StrongPass123!';

* Chức năng của Contained User trong SQL server
- Toàn bộ thông tin xác thực (username + password) nằm 100% bên trong database.
- Khi connect, SQL Server không kiểm tra Login ở master database nữa → Authentication + Authorization đều diễn ra chỉ trong database.
- Database trở thành self-contained: Bạn restore/attach/migrate sang server khác mà không cần tạo lại Login + mapping (không còn orphan user).

Lưu ý quan trọng trong SQL:
- Muốn xem thông tin hay truy vấn đến một database nào đó thì trước hết cần phải "USE" database đó.

check trustworthy DB
SELECT name, is_trustworthy_on FROM sys.databases;
TRUSTWORTHY = ON nghĩa là SQL Server tin tưởng database đó, cho phép nó thực hiện hành động vượt ra ngoài phạm vi database
Thuộc tính Trustworthy (Đáng tin cậy) của cơ sở dữ liệu giúp xác định xem SQL Server có tin tưởng (trust) vào cơ sở dữ liệu đó hay không.
Khi làm việc với CLR (Common Language Runtime – mã .NET chạy bên trong SQL Server), sẽ có rất nhiều trường hợp mà các lệnh đặc biệt hoặc stored procedure yêu cầu phải có quyền hạn (privileges) cao. Lúc này, thuộc tính Trustworthy cần được bật để SQL Server cho phép thực thi những mã đó. Nó hoạt động như một “giấy phép” giúp bảo vệ cơ sở dữ liệu khỏi các tình huống độc hại (malicious scenarios).
Có nhiều thuộc tính khác nhau trên Windows Server và SQL Server để kiểm tra xem cơ sở dữ liệu có được tin tưởng hay không. Những thuộc tính này phải được thiết lập đúng cách thì SQL Server mới hoạt động bình thường. Một trong những cách phổ biến là sử dụng lệnh TRUSTWORTHY trên cả hai server.

* Linked Server là tính năng cho phép một instance SQL Server (local) có thể kết nối và thực thi query/command trực tiếp trên một instance khác (remote) như thể đang làm việc trên local.
Nó không chỉ là kết nối đơn thuần mà là cầu nối tin cậy (trust link) giữa các server

Liệt kê toàn bộ linked server đã được định nghĩa trên instance hiện tại (local server)
EXEC sp_linkedservers;
Hữu ích:
Đây là lệnh nhanh nhất để biết “có linked server nào không?”.
Giúp bạn biết remote server ở đâu (IP/hostname) và loại provider → biết có thể khai thác gì (SQL-to-SQL, SQL-to-Oracle…).
Nếu không có linked server nào → bỏ qua phần này. Nếu có → tiếp tục enumerate mapping (rất nguy hiểm).

lệnh liệt kê linked server, giống lệnh trên nhưng nó là lệnh mới nên có nhiều thông tin giá trị hơn
SELECT * FROM sys.servers;

Gửi một query pass-through (chạy hoàn toàn trên remote server) qua linked server và trả kết quả về local. Nó thực hiện test connection, biết được phiên bản, linked server có hoạt động hay không, current user trên máy remote nếu thay version thành system_user..., nói chung là nó thực hiện lệnh trên remote và kết quả sẽ được trả về local.
SELECT * FROM OPENQUERY([LinkedServerName], 'SELECT @@version');


EXEC ('SELECT @@version') AT [LinkedServerName];
Đây là lệnh mạnh nhất trong enumeration (có thể dẫn đến RCE).
Gửi toàn bộ batch T-SQL (có thể nhiều lệnh) để chạy trên remote server như thể bạn đang login trực tiếp vào remote.
Khác biệt lớn với OPENQUERY: EXEC AT cho phép dynamic SQL, chạy nhiều lệnh, và thực thi extended procedure (xp_cmdshell, sp_addlogin…).

So sánh nhanh OPENQUERY vs EXEC AT (rất quan trọng trong pentest):
OPENQUERY → chỉ đọc dữ liệu (recon).
EXEC AT → thực thi lệnh (attack, escalation, RCE).

- sys.* là gì? Trong Microsoft SQL Server thì sys = schema hệ thống chứa các system catalog views (view metadata). sys không phải database, sys là schema chứa các view hệ thống
+ sys dùng để: Xem cấu trúc database
Xem user, quyền
Xem table, column
Xem config hệ thống
+ nó tồn tại ở mọi db, mỗi db sẽ có mỗi view khác nhau. ví dụ ở db quyền cao thì có thể xem được nhiều thông tin hơn.
| View                    | Chỉ dùng ở đâu                        |
| ----------------------- | ------------------------------------- |
| `sys.server_principals` | server-level (thường query từ master) |
| `sys.databases`         | toàn server                           |
Có thể query từ DB khác, nhưng: vẫn là thông tin toàn server
+ sys.* nó phụ thuộc vào context db hiện tại
- Vì sao nó phụ thuộc context db hiện tại mà có thể query toàn server?
+ Vì có 2 loại sys views khác nhau:
	+ Database-level views (phụ thuộc context)
| View                      | Phạm vi       |
| ------------------------- | ------------- |
| `sys.tables`              | DB hiện tại   |
| `sys.columns`             | DB hiện tại   |
| `sys.database_principals` | user trong DB |
	+ Server-level views (KHÔNG phụ thuộc DB)
| View                    | Phạm vi        |
| ----------------------- | -------------- |
| `sys.databases`         | toàn bộ server |
| `sys.server_principals` | tất cả login   |
| `sys.servers`           | linked server  |
là global metadata, không phụ thuộc riêng  db nào
